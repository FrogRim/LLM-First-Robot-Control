## 1. 치명적 지적 사항 (Critical)

### 1.1 ⚠️ 데이터 규모의 타당성 문제

**교수님 예상 질문**:
> "350개 샘플로 14B 파라미터 모델을 fine-tuning하는 것이 과연 통계적으로 유의미한가? 과적합 우려는 없는가?"

**현재 상황**:
- 학습 데이터: 297개 (train)
- 평가 데이터: 53개 (test)
- 모델 크기: Qwen2.5-14B (140억 파라미터)
- 학습 가능 파라미터: LoRA로 약 2,000만 개 (0.14%)

**문제점**:
1. **데이터/파라미터 비율 불균형**
   - 전통적 지침: 파라미터당 최소 10-100개 데이터 필요
   - 현재: 297 samples / 20M params = 0.00001485 (매우 낮음)
   - QLoRA 사용으로 완화되지만, 여전히 적은 편

2. **테스트 셋 크기 부족**
   - 53개 샘플로는 통계적 신뢰도 확보 어려움
   - 재료당 평균 7.5개 샘플 (부족)

**대응 방안**:
- LoRA의 parameter-efficient 특성 강조
- "도메인 적응(domain adaptation)"이지 "full fine-tuning"이 아님을 명확히
- Transfer learning 맥락에서 적절한 규모임을 논증
- 실제로 과적합 없음을 학습 곡선으로 입증 (train/eval loss 비율 1.17)

**보완 계획**:
```
현재: 350개 → 목표: 500-1,000개 (Week 1)
- DROID 추가 에피소드 활용
- 더 다양한 작업 유형 포함
```

### 1.2 ⚠️ 추론 시간 목표 미달성

**교수님 예상 질문**:
> "연구 목표에서 '실시간 제어'를 언급했는데, 15.6초는 실시간이라고 할 수 없지 않은가?"

**현재 상황**:
- 목표: <200ms (0.2초)
- 달성: 15,567ms (15.6초)
- **격차**: 77배 느림

**문제점**:
1. **목표 설정의 비현실성**
   - 14B 모델로 200ms 달성은 현재 하드웨어로 거의 불가능
   - A100 GPU에서도 어려운 수치

2. **"실시간 제어" 용어의 부적절성**
   - 로봇 제어에서 실시간(real-time)은 보통 100ms 이하
   - 현재 시스템은 "near real-time" 또는 "offline planning"에 가까움

**대응 방안**:
1. **목표 재정의**:
   ```
   기존: "실시간 제어 가능성 검증"
   수정: "준실시간 제어 파라미터 생성 시스템"
   ```

2. **Two-Stage 아키텍처 제안**:
   - Stage 1 (Offline): LLM으로 물리 속성 추론 (15초)
   - Stage 2 (Online): 캐시된 속성으로 빠른 제어 생성 (0.4ms)
   - 동일 물체 반복 조작 시 Stage 2만 사용

3. **벤치마크 비교**:
   - 유사 연구의 추론 시간 조사 및 비교
   - 현재 시스템이 실용적 범위 내임을 입증

**보완 계획**:
- LoRA 병합 후 10초 이하 달성 예상
- 더 작은 모델(7B) 증류 가능성 언급
- 실제 응용에서는 물체별 1회 추론 후 캐싱 전략

**현재 상황**:

- 하지만 ground truth가 없음
- 명령어에 재료명이 명시되어 있어 쉬운 문제

**문제점**:
1. **순환 논리 (Circular Reasoning)**
   - 입력: "Pick up the **plastic** bottle"
   - 출력: material_inference: "plastic"
   - 이것은 추론이 아니라 단순 추출

2. **실제 물리 속성 검증 없음**

**대응 방안**:
1. **평가 방법 수정**:
   - 재료명이 없는 명령어로 테스트 필요
   - 예: "Pick up the transparent cup" → glass 추론해야 함

2. **Ground Truth 확보 방안**:
   - DROID 데이터셋의 물체 메타데이터 활용
   - 문헌 조사를 통한 표준 제어 파라미터 비교

3. **정성적 평가 추가**:
   - Genesis AI 시뮬레이션에서 실제로 물체를 들어올리는지 확인

### 2.3 🟡 학습 손실의 절대값 해석

**교수님 예상 질문**:
> "최종 손실 0.278이 좋은 것인가 나쁜 것인가? 비교 기준이 없다."

**현재 상황**:
- 최종 train loss: 0.278
- 최종 eval loss: 0.325
- "목표 0.5 이하 달성"이라고 하지만, 0.5의 근거 불명확

**문제점**:
1. **손실 값의 의미 불명확**:
   - Cross-entropy loss인가, MSE인가?
   - 0.278이 domain-specific하게 좋은 것인지 판단 어려움

2. **목표 설정 근거 없음**:
   - "목표 0.5"이 어디서 나온 값인지 불명확

**대응 방안**:
1. **손실 함수 명시**:
   - 사용한 손실 함수 종류 명확히 기술
   - Causal Language Modeling loss (cross-entropy)

2. **상대적 개선 강조**:
   - Epoch 1 (1.234) → Epoch 4 (0.278): 77% 감소
   - Base model과 비교 시 상대적 개선 측정

3. **Perplexity 등 해석 가능 지표 추가**:
   - Perplexity = exp(loss) = exp(0.278) ≈ 1.32
   - "모델이 평균 1.32개의 토큰 후보 중 선택" → 높은 확신도
